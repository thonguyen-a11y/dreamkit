# DreamKit Constitution

## Core Principles

### I. Feature-First, Domain-Driven Architecture
Code MUST be organized by feature/domain, not by technical layer. Each domain
(e.g., `catalogue`, `cart`, `checkout`, `auth`, `account`, `admin`) owns a
self-contained module exposing its own components, hooks, state, API access,
and types; cross-domain reuse happens only through explicit, published
interfaces — never by reaching into another domain's internals.

The Dependency Rule MUST hold in every module: dependencies point inward,
from UI → application/hooks → domain logic → data access, and never the
reverse. Domain and application logic MUST NOT import from `app/` (routing)
or from UI component internals. Shared, domain-agnostic code (generic UI
primitives, utilities, the API client) lives in a common layer (`components/`,
`lib/`) that any domain may depend on, but which MUST NOT depend back on any
specific domain.

**Rationale**: Feature-first, domain-driven boundaries with a one-directional
dependency rule keep the codebase navigable as the catalogue, cart, checkout,
auth, and admin surfaces grow independently, prevent circular coupling, and
let any one domain be refactored, tested, or extracted without cascading
changes through the rest of the app.

### II. Performance Budget & Lazy-Loading by Default
Every shippable route MUST meet, at p75 on mid-tier mobile hardware:
LCP < 2.5s, INP < 200ms, and CLS < 0.1. Each route's initial (critical-path)
JavaScript payload MUST stay within budget — 170KB gzipped for standard
storefront routes, 250KB gzipped for the admin area — measured via the
Next.js build output; a route that exceeds budget MUST be split further or
justified explicitly in the PR description.

Code that is not required for the initial paint or above-the-fold
interaction MUST be lazy-loaded by default: use `next/dynamic` for
below-the-fold sections, modals, admin widgets, and any dependency heavier
than ~20KB gzipped; use React Server Components and streaming (`loading.tsx`,
`Suspense`) instead of client-side fetching where the data doesn't need
interactivity; images MUST use `next/image`, and fonts MUST be self-hosted
and subset via `next/font`.

**Rationale**: Shopping traffic is disproportionately mobile and
conversion-sensitive to load speed; a numeric, enforced budget and a
lazy-by-default rule stop performance regressions from accumulating silently
one "small" import at a time.

### III. SEO & Discoverability
Every public, indexable route MUST export metadata via the Next.js Metadata
API (`generateMetadata` for dynamic routes), including title, description,
canonical URL, and Open Graph/Twitter Card tags. Product and category pages
MUST additionally emit JSON-LD structured data (`Product`, `Offer`,
`BreadcrumbList`, and `Organization` where applicable) reflecting the actual
rendered price, availability, and imagery. Canonical URLs MUST be absolute,
self-referencing on the primary route, and MUST point to the canonical
(non-filtered, non-paginated-duplicate) variant of a listing page.
`sitemap.xml` and `robots.txt` MUST stay in sync with published, indexable
routes.

**Rationale**: For a storefront, organic discovery and correct rich-result
rendering in search directly drive traffic and revenue; metadata and
structured data are treated as a shipping requirement, not an afterthought.

### IV. Observability (Logging, Error Monitoring, Analytics)
All server-side request handlers, API routes, and Server Actions MUST emit
structured (JSON) logs with a correlation/request ID, and MUST NOT log
secrets, credentials, or raw PII. Unhandled exceptions on both client and
server MUST be captured by an integrated error-monitoring service (e.g.,
Sentry or equivalent) with source maps uploaded so stack traces resolve to
original TypeScript. Key business events (product view, add-to-cart, checkout
start/complete, sign-up, sign-in) MUST be emitted to an analytics pipeline
with a consistent event schema.

**Rationale**: A storefront that fails silently loses revenue and trust;
logging, error monitoring, and analytics are the minimum instrumentation
needed to detect, diagnose, and measure the impact of production issues
before users report them.

### V. CI/CD Quality Gates
No change MAY be merged to the default branch unless the CI pipeline passes,
in order: type checking (`tsc --noEmit`), linting (`eslint`), the automated
test suite (`vitest run`), and a production build (`next build`). Any one of
these failing MUST block the merge; gates MUST NOT be skipped, disabled, or
weakened to unblock a merge without an explicit, reviewed exception recorded
in the PR.

**Rationale**: These four gates are the cheapest, fastest way to catch
regressions (type errors, style/correctness lint violations, broken
behavior, broken builds) before they reach users, and making them mandatory
removes any ambiguity about what "ready to merge" means.

## Technology Stack

The storefront is built on Next.js 16 (App Router) with React 19 and
TypeScript in strict mode. Server state (data fetched from APIs) MUST be
managed with React Query; client/UI state that is not server data (cart
drawer open/closed, filter selections, wizard steps, etc.) MUST be managed
with Zustand — server state MUST NOT be duplicated into Zustand stores, and
Zustand state MUST NOT be used to cache server responses. Styling MUST use
Tailwind CSS; UI primitives MUST be built on shadcn/ui components rather than
introducing a competing component library, unless a documented gap in
shadcn/ui requires a one-off exception. Any deviation from this stack (a new
state library, a second CSS approach, a competing component kit) MUST be
justified in the PR and, if it recurs, promoted into this constitution via
an amendment.

## Development Workflow

Every feature or bugfix branch is expected to map to a feature-first module
boundary as described in Principle I. Pull requests MUST state which
principle(s) are relevant to the change and call out any deliberate
deviation (e.g., a route intentionally over the JS budget, a page without
structured data because it is `noindex`) with a one-line justification.
Reviewers MUST verify the CI gates in Principle V have passed before
approving.

## Governance

This constitution supersedes ad hoc conventions and prior undocumented
practice for this repository. Where code and constitution conflict, the
constitution wins unless amended.

**Amendment procedure**: Amendments are made by editing this file directly,
updating the Sync Impact Report at the top of the file, and bumping the
version per the policy below. Any amendment touching Core Principles MUST
also review `.specify/templates/plan-template.md`, `spec-template.md`, and
`tasks-template.md` (once they exist in this repo) for consistency, and
MUST record any file left out of sync as a follow-up TODO in the Sync Impact
Report.

**Versioning policy** (semantic versioning applied to governance):
- MAJOR: backward-incompatible removal or redefinition of a principle.
- MINOR: a new principle or materially expanded section is added.
- PATCH: wording, clarification, or typo fixes with no semantic change.

**Compliance review**: Every `/speckit-plan` MUST include a Constitution
Check gate; any plan that cannot satisfy a principle MUST document the
deviation and its justification explicitly rather than silently ignoring the
principle. Complexity or scope that violates Principle I or II without
justification MUST be simplified before implementation proceeds.
